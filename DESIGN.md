# Design: Composer, a Track-First Windows DAW

## Problem Statement

취미 작곡가는 피아노, 기타, 베이스, 드럼과 녹음을 한 프로젝트에서 다루고 싶지만 Cubase급 DAW는 처음부터 너무 많은 기능과 패널을 노출한다. 반대로 단순한 코드 진행 도구는 일반적인 트랙 편집과 실제 녹음으로 자연스럽게 확장되지 않는다.

Composer는 일반 DAW의 타임라인과 트랙 관리 방식을 유지하면서, 각 악기 트랙 안에서 코드 음표를 빠르게 삽입하고 드래그로 멜로디를 수정할 수 있게 한다. 첫 사용자가 10분 안에 밴드 편성의 1절을 만들 수 있어야 하지만, 녹음과 트랙 추가는 작업 어느 시점에서나 가능해야 한다.

## What Makes This Cool

- 밴드 부원을 추가하듯 `+ Add instrument`로 악기 트랙을 자유롭게 만든다. 피아노, 기타, 베이스, 드럼은 기본 제공 악기일 뿐 고정 슬롯이 아니다.
- MIDI 클립 안에서 `Ctrl+K`를 누르면 루트, 코드 종류, 전위, 음역을 고르고 해당 음들이 평범한 MIDI 노트로 삽입된다.
- 삽입된 코드 음은 즉시 이동, 길이 조절, 삭제, 분산화음 편집이 가능하다. 사용자를 특수한 코드 객체에 가두지 않는다.
- 코드 음을 그대로 유지하거나 `Generate rhythm from notes`로 악기별 리듬을 미리 듣고 적용할 수 있다.
- 보컬과 외부 악기 오디오 트랙은 처음부터 추가하고 녹음할 수 있다. “10분 1절”은 안내 목표이지 강제 단계가 아니다.

## Constraints

- 첫 배포 대상은 Windows x64 데스크톱이다.
- 오디오 엔진과 앱 UI는 C++20, CMake, JUCE를 사용한다.
- 대상 사용자는 코드 이름은 알 수 있지만 전문 DAW 지식은 많지 않은 취미 작곡가다.
- 기본 제공 악기는 피아노, 기타, 베이스, 드럼이며 신스 등 추가 악기 팩을 등록할 수 있어야 한다.
- 첫 릴리스는 VST3 호스팅, 피치 보정, 테이크 컴핑, 악보 편집을 포함하지 않는다.
- 프로젝트 편집은 비파괴 방식이어야 하며 모든 사용자 명령은 Undo/Redo를 지원한다.

## Premises

1. 트랙 관리, 재생 헤드, 클립, 믹서 개념은 기존 DAW 관습을 따른다.
2. “밴드 부원”은 트랙 추가를 쉽게 설명하는 UI 은유이며 별도 도메인 객체가 아니다.
3. 코드 기능은 전역 코드 진행 트랙이 아니라 선택한 MIDI 트랙의 편집 도구다.
4. 코드 삽입 결과는 독점 객체가 아닌 일반 MIDI 노트 이벤트다.
5. 오디오 녹음은 편곡 전, 중, 후 언제든 가능하며 MIDI 작업과 동일한 타임라인에 놓인다.
6. 쉬운 화면은 기능 삭제가 아니라 상황별 노출과 점진적 공개로 만든다.

## Approaches Considered

### Approach A: Electron + Web Audio

UI 프로토타입은 빠르지만 Windows 오디오 장치, 낮은 지연 모니터링, ASIO 중심 녹음에서 위험이 크다. 제품의 가장 어려운 기반을 나중으로 미루므로 채택하지 않았다.

### Approach B: Native C++ + JUCE

오디오/MIDI 장치, 실시간 처리, 녹음, 파일 입출력, DSP와 UI를 한 애플리케이션에서 관리한다. UI 개발 속도는 느리지만 장기적인 DAW 기반으로 가장 안정적이어서 채택했다.

### Approach C: JUCE Engine + Tauri/React UI

UI 반복 속도는 빠르지만 프론트엔드와 실시간 엔진 사이의 상태 동기화, 장애 분석, 배포가 복잡하다. 초기 제품에는 과도한 구조다.

## Recommended Approach

### Architecture

```text
JUCE Application
├── App Shell
│   ├── Transport / Tempo / Metronome
│   ├── Track List / Timeline / Mixer
│   └── Context Editor: Piano Roll or Waveform
├── Project Model
│   ├── InstrumentTrack -> MidiClip -> NoteEvent
│   ├── AudioTrack -> AudioClip -> SourceFile
│   └── TempoMap / TimeSignature / Markers
├── Editing Services
│   ├── CommandManager + Undo/Redo
│   ├── ChordInsertionService
│   └── RhythmGenerationService
├── Audio Engine
│   ├── Transport / Scheduler / Metronome
│   ├── Instrument and Track Processor Graph
│   ├── Audio and MIDI Device Manager
│   └── Recorder / Offline WAV Renderer
└── Persistence
    ├── Versioned Project Manifest
    ├── Relative Audio Assets
    └── Autosave / Crash Recovery
```

UI와 오디오 엔진은 같은 네이티브 프로세스에 두되 경계를 분리한다. UI 스레드는 명령과 불변 스냅샷으로 상태를 교환하고, 오디오 콜백에서는 파일 접근, 메모리 할당, 락 획득을 금지한다.

### Project Model

- `Project`: 샘플레이트, 템포 맵, 박자표, 트랙 순서, 루프 범위, 마커를 가진다.
- `InstrumentTrack`: 이름, 색상, 음량, 팬, mute/solo/arm, 악기 프리셋과 MIDI 클립을 가진다.
- `AudioTrack`: 동일한 트랙 제어와 입력 라우팅, 모니터링 모드, 오디오 클립을 가진다.
- `MidiClip`: 시작, 길이, 루프와 일반 `NoteEvent { pitch, start, duration, velocity }`를 가진다.
- `AudioClip`: 원본 파일을 수정하지 않고 위치, trim, gain, fade, stretch-disabled 상태를 저장한다.
- 프로젝트 파일은 버전이 붙은 JSON manifest와 `Audio/`, `Autosave/` 폴더로 구성한다. MIDI와 편집 상태는 manifest에, 녹음 파일은 WAV로 저장한다.

### Standard Track Workflow

1. 빈 프로젝트 또는 “Band sketch” 템플릿으로 시작한다.
2. `+ Add instrument`에서 악기와 프리셋을 골라 트랙을 추가한다.
3. `+ Add audio track`에서 입력 장치를 정하고 보컬 또는 외부 악기 트랙을 추가한다.
4. 트랙은 드래그 재정렬, 이름 변경, 복제, 삭제가 가능하다.
5. 각 헤더는 mute, solo, record-arm, volume, pan을 제공하며 세부 라우팅은 펼쳤을 때만 보인다.
6. MIDI 클립을 더블클릭하면 피아노 롤, 오디오 클립을 더블클릭하면 파형 편집기가 같은 하단 영역에 열린다.

### Chord Shortcut

1. 사용자가 pitched instrument의 MIDI 클립에서 삽입 위치를 선택하고 `Ctrl+K`를 누른다.
2. 팝오버에서 root, quality, extension, inversion, octave/range를 선택한다.
3. 현재 스케일 밖의 코드에는 경고만 표시하고 삽입을 막지 않는다.
4. 미리듣기는 현재 트랙의 악기로 재생한다.
5. `Insert notes`는 코드 구성음을 그리드 길이로 일반 MIDI 노트에 삽입한다. 한 번의 Undo 명령으로 묶는다.
6. 기타·베이스 프리셋은 연주 가능한 음역과 기본 보이싱을 제안하지만 사용자가 모든 음을 수정할 수 있다.
7. 드럼과 오디오 트랙에서는 코드 도구를 숨긴다. 선택이 없거나 클립이 잠겼다면 실행 이유를 짧게 설명한다.

### Rhythm Generation

- 코드 노트를 선택하고 `Generate rhythm from notes`를 실행한다.
- 패턴은 sustained, block rhythm, arpeggio, guitar strum, bass pulse처럼 악기 역할별로 제공한다.
- 원본을 보존한 채 임시 미리듣기를 제공하고 `Apply`할 때 일반 MIDI 노트로 확정한다.
- seed를 저장해 같은 설정으로 같은 결과를 재생성할 수 있다.
- 적용 전체가 하나의 Undo 명령이며 생성 후에도 음표를 직접 편집할 수 있다.

### Recording

- 어느 시점에서나 오디오 또는 MIDI 트랙을 record-arm하고 녹음할 수 있다.
- 입력 장치와 채널 선택, 입력 레벨 미터, clipping 경고, direct/software monitoring 선택, count-in, metronome을 제공한다.
- Windows 장치는 WASAPI를 기본으로 사용하고 사용 가능한 환경에서는 ASIO를 선택할 수 있게 한다.
- 녹음 시작 전에 쓰기 가능 경로와 남은 디스크 공간을 확인한다. 녹음 중 파일은 스트리밍 저장하고 완료 후 프로젝트에 원자적으로 등록한다.
- round-trip latency를 측정 또는 사용자가 보정할 수 있으며, 기록된 클립 위치에 보정값을 적용한다.
- 장치 분리, 샘플레이트 변경, 디스크 부족, 입력 dropout 발생 시 녹음 파일을 보존하고 복구 가능한 오류를 표시한다.
- 첫 릴리스 파형 편집은 이동, trim, split, fade, gain, crossfade를 지원한다. 피치 보정과 멀티테이크 컴핑은 후속 범위다.

### Instruments

- 기본 피아노, 기타, 베이스, 드럼은 내장 샘플 기반 악기로 제공한다.
- 악기 레지스트리는 `instrumentId`, 표시 이름, 역할, 음역, 프리셋, 샘플 팩 위치를 가진다.
- 사용자는 같은 악기를 여러 번 추가하거나 제거할 수 있고 프로젝트에 고정 트랙 수 제한을 두지 않는다.
- 샘플 팩 누락 시 프로젝트는 열리며 해당 트랙을 offline으로 표시하고 재연결을 제공한다.
- VST3 악기/효과 호스팅은 오디오 그래프 경계는 마련하되 첫 릴리스 뒤로 미룬다.

### Interaction Design

- 상단 transport는 재생, 정지, 녹음, tempo, 박자, metronome, loop만 항상 노출한다.
- 중앙은 일반적인 트랙 타임라인이며 왼쪽 track header와 오른쪽 clip lane으로 구성한다.
- 하단 contextual editor는 선택한 클립에 필요한 도구만 표시한다.
- 음표 드래그는 pitch/time 이동, 좌우 가장자리 드래그는 길이, 더블클릭은 추가, Delete는 삭제다.
- grid snap은 기본 활성화하고 `Alt`를 누르는 동안만 일시 해제한다.
- 모든 위험 동작은 즉시 Undo 가능해야 하며 자동 생성은 적용 전 미리듣기를 제공한다.
- 승인된 와이어프레임: [`.gstack-daw-wireframe.png`](.gstack-daw-wireframe.png)

### Mixer and Effects

- 기본 화면에서는 track header의 volume, pan, mute, solo와 peak meter만 노출한다.
- 사용자가 Mixer를 열면 각 트랙의 channel strip, master bus, input/output routing을 표시한다.
- 첫 릴리스는 track gain/pan, 3-band EQ, compressor, reverb send, master limiter를 기본 효과로 제공한다.
- 효과 순서와 파라미터는 프로젝트에 저장하며 bypass와 Undo/Redo를 지원한다.
- automation lane과 서드파티 effect hosting은 후속 범위로 미룬다.

## Edge Cases

- 프로젝트 로드 시 사용하던 오디오/MIDI 장치가 없음
- 클립 경계를 넘는 chord note 또는 음수 위치 삽입
- 서로 다른 템포 구간에 걸친 MIDI/오디오 클립
- 녹음 중 tempo 변경, loop recording, transport seek 시도
- 같은 입력을 여러 armed track이 공유하는 경우
- software monitoring으로 feedback 가능성이 높은 라우팅
- 프로젝트 저장 중 앱 종료 또는 시스템 절전
- 오디오 파일 이동, 손상, sample pack 누락
- 수천 개 노트 선택 후 이동 또는 rhythm generation
- Undo 중 재생 중이거나 녹음 중인 경우

## Success Criteria

- 대상 사용자 5명 중 4명이 도움 없이 10분 안에 악기 4개를 추가하고 8마디 1절을 재생한다.
- 사용자가 임의의 MIDI 트랙에서 5초 안에 코드 도구를 열고 코드 음표를 삽입한다.
- 삽입된 각 음표를 드래그, resize, delete하고 한 번에 Undo할 수 있다.
- 작곡 도중 오디오 트랙을 추가하고 count-in 후 30초 녹음을 완료한다.
- 48 kHz 프로젝트에서 32개 트랙, 16개 동시 내장 악기, 256-sample buffer로 30분 재생 시 dropout이 없다.
- 저장 후 다시 열었을 때 트랙, 클립, 악기, 편집, 녹음 파일 참조가 보존된다.
- 3분 프로젝트를 WAV로 내보낸 결과가 실시간 재생과 sample-accurate하게 일치한다.
- 강제 종료 후 마지막 autosave가 감지되고 손상되지 않은 녹음 파일을 복구한다.

## Test Plan

- 단위 테스트: 코드 구성음, extension, inversion, 음역 보정, grid 위치, Undo transaction.
- 속성 테스트: 모든 root/quality/inversion 조합이 유효한 MIDI pitch와 양수 duration을 생성한다.
- 단위 테스트: tempo map 변환, clip trim/split, project migration, asset path 정규화.
- 오프라인 오디오 테스트: metronome, scheduler, built-in instrument render를 golden WAV와 비교한다.
- 통합 테스트: MIDI 녹음, 오디오 녹음, 장치 분리, 디스크 부족, sample-rate mismatch, crash recovery.
- UI 테스트: 트랙 추가/삭제/재정렬, chord popup, piano-roll drag/resize, waveform edit, Undo/Redo.
- 믹서 테스트: routing, mute/solo, built-in effect order, project reload, master clipping prevention.
- 성능 테스트: 32-track project 재생과 30분 record/playback soak test.
- 수동 장치 매트릭스: WASAPI shared/exclusive, ASIO, USB mic, 2-channel interface, MIDI keyboard.

## Distribution Plan

- CMake preset으로 Windows x64 Release 빌드를 재현한다.
- GitHub Actions Windows runner에서 format/lint, unit/integration test, Release build, smoke launch를 실행한다.
- 버전이 붙은 서명된 installer와 SHA-256 checksum을 GitHub Releases에 게시한다.
- installer는 앱, 기본 악기 팩, 제거 프로그램을 포함하고 사용자 프로젝트를 제거하지 않는다.
- code signing 인증서는 CI secret으로만 접근하며 서명 실패 시 배포를 중단한다.
- 업데이트는 첫 릴리스에서 수동 다운로드로 시작하고 검증된 manifest 기반 updater는 후속으로 추가한다.

## Dependencies

- JUCE와 CMake toolchain

- Windows code-signing certificate
- 실제 WASAPI/ASIO 오디오 인터페이스와 MIDI keyboard가 있는 테스트 환경
- UI 상세 설계를 위한 `/plan-design-review`, 실행 계획을 위한 `/plan-eng-review`

### Section 1: Architecture Review

The chosen boundaries are directionally sound, but the original plan did not specify how mutable UI state reaches the real-time graph. The fix is an immutable render snapshot plus a lock-free command queue; graph rebuilds occur off the audio callback and are swapped at a buffer boundary. Persistence owns durable state, the audio engine consumes compiled snapshots, and neither layer reaches through the other.

```text
UI Thread                 Model Thread                 Audio Thread
commands/gestures --> CommandDispatcher --> ProjectModel
                           |                    |
                           +--> SnapshotBuilder +--> atomic snapshot swap
                           |                    |
                           +--> Persistence     +--> scheduler/process graph --> device

Recorder writer thread <-- bounded audio FIFO <-- audio callback
Waveform worker         <-- committed audio file
```

The architecture scales by track count and voice count, not network load. At 10x, instrument voices and waveform caches fail first; processor nodes therefore declare cost, tracks can freeze later, and waveform peaks are tiled. The single points of failure are the audio device, recording destination, project manifest, and sample pack registry; each now has an explicit degraded state.

### Section 2: Error And Rescue Registry

Typed domain errors cross thread boundaries as data. Exceptions never escape through the audio callback.

| Codepath | Error type | Rescued | Rescue action | User sees |
|---|---|---:|---|---|
| open audio device | `AudioDeviceOpenError` | Yes | keep project open, offer device picker and safe defaults | “장치를 열 수 없습니다” + cause + fix |
| configure sample rate | `UnsupportedSampleRateError` | Yes | propose supported rate, do not mutate project silently | supported choices |
| start recording | `RecordingPathUnavailable` | Yes | block start, choose writable folder | exact path and remediation |
| recording preflight | `DiskSpaceLowError` | Yes | block or shorten requested take | remaining minutes estimate |
| recording stream | `AudioWriteError` | Yes | stop safely, finalize recoverable partial file | saved partial-file location |
| active device | `DeviceDisconnectedError` | Yes | stop transport, preserve file, reopen picker | disconnected device name |
| load project | `ProjectParseError` | Yes | open recovery/autosave chooser, never overwrite source | parse location and recovery action |

### Section 3: Security And Threat Model

This is a local application, so authorization is not the main risk. File parsing, path traversal, signed distribution, sample-pack provenance, and future plugin loading are. Project manifests use a strict versioned schema, maximum counts and string lengths, canonicalized paths, and explicit external-asset consent; imported MIDI/audio is decoded through bounded readers and never used to construct shell commands.

| Threat | Likelihood | Impact | Mitigation |
|---|---|---|---|
| malicious project path escapes project directory | Medium | High | canonicalize, reject traversal, explicit external references only |
| malformed audio/MIDI causes OOM or hang | Medium | High | size/count/duration limits, fuzz tests, cancellable background decode |
| tampered installer/sample pack | Medium | High | code signing, SHA-256 manifest, trusted HTTPS release source |
| secrets leak from CI | Low | High | signing credential only in CI secret store, least privilege, rotation runbook |
| diagnostic bundle contains audio or paths | Medium | Medium | metadata-only default, preview/redact before export, no automatic upload |
| third-party plugin compromise | N/A in v1 | High | VST3 deferred until isolation and trust model exist |

### Section 4: Data Flow And Interaction Edge Cases

```text
Chord input --> validate --> preview --> one Undo transaction --> NoteEvents --> snapshot
  missing       invalid      cancelled      conflict/locked clip      audio playback
  no clip       range        audition fail  generation stale          visible result

Record arm --> preflight --> count-in --> FIFO writer --> finalize --> AudioClip
  no input      low disk      cancel         device loss     write err   waveform build

Open project --> schema check --> migrate copy --> resolve assets --> compile snapshot
  missing file    malformed      newer ver       offline refs      safe playback
```

Double activation is idempotent: record and render commands carry operation IDs, and a second in-flight command is rejected visibly. Transport seek, tempo edit, graph edit, save, and undo have explicit policies while recording; destructive graph mutations queue until the take stops, while safe mixer changes apply at a buffer boundary. Empty MIDI clips, zero-length audio, notes beyond clip bounds, stale rhythm previews, and duplicate input routing are covered by validation and user-visible recovery.

### Section 7: Performance Review

At 48 kHz and 256 samples the callback deadline is 5.33 ms. The reference project target is callback p99 below 60% of that budget, zero allocation, zero blocking locks, zero disk I/O, and no diagnostic formatting on the audio thread. FIFO overflow increments an atomic counter and triggers a UI warning outside the callback.

The three likely slow paths are polyphonic sample playback, waveform peak generation, and offline render. Samples are memory-mapped or streamed with bounded caches, waveform peaks are tiled and cancellable, and offline render reports progress without blocking the UI. Performance tests record callback duration, underruns, peak memory, load time, save time, and export real-time factor on a named reference machine.

### Section 8: Observability And Debuggability Review

Remote telemetry is not required for v1. Local structured logs, a bounded in-memory real-time health ring, device configuration snapshots, and an explicit “Export diagnostics” action are required. The bundle excludes audio content and redacts user paths by default.

Day-one counters are device-open failures, callback underruns, FIFO high-water mark, recording write failures, project migration failures, asset relink count, render failures, and crash-recovery success. Every logged operation carries project-session and operation IDs; the runbook maps each typed error to user action and diagnostic fields.

### Section 9: Deployment And Rollout Review

There is no server rollout. Release safety depends on reproducible Windows builds, installer signing, project-format compatibility, and rollback. CI builds on a pinned toolchain, runs unit/integration/offline-render tests, launches a smoke project, signs artifacts, and publishes checksums only after every gate passes.

```text
tag -> clean CI build -> tests -> installer smoke VM -> sign -> checksum -> release
                                                         |
upgrade -> backup project before migration -> open copy -> verify -> commit new format
   failure -> retain original -> reinstall prior signed version -> open original/read-only
```

The installer never removes projects or user sample packs. Every migration writes a new temporary manifest, validates it, keeps a backup, then atomically replaces the active manifest. A prior signed installer remains available for rollback.

### Section 10: Long-Term Trajectory Review

Reversibility is 4/5 if project schema, command boundaries, and processor graph remain stable. JUCE licensing, project file semantics, and sample-pack rights are the least reversible choices and must be resolved before the first code milestone. VST3, automation, and additional instruments can attach to the processor and parameter boundaries without changing the track model.

The six-month regret scenario is shipping many nominal features with weak audio reliability and low-quality instruments. The prevention is milestone exit gates: no phase advances until the acceptance fixture plays, saves, recovers, and renders deterministically. Future features consume the same public domain commands instead of reaching into engine internals.

The approved wireframe establishes a conventional DAW hierarchy: transport first, tracks and timeline second, contextual editor third. It does not yet define a design system, empty/error states, compact-window behavior, screen-reader semantics, or focus order. Phase 2 must add those details before UI implementation.

```text
Launch
  +-> Empty project -> Add instrument/audio -> Timeline editing
  |                                          +-> MIDI editor -> chord/rhythm
  |                                          +-> Wave editor -> record/edit
  +-> Template -> prefilled band tracks -----+
```

Windows desktop is intentional; mobile is not a target. Responsive behavior means defined minimum window size, compact mode, panel collapse priority, keyboard navigation, 44 px targets for primary controls, visible focus, and WCAG 2.2 AA contrast.

### Time And Identity Model

- Every entity uses a stable UUID persisted in the manifest. Array position is never identity.
- MIDI and musical clip positions use signed 64-bit ticks at 960 PPQ.
- The transport and renderer use signed 64-bit project sample positions.
- `TempoMap` is the only ticks/samples conversion authority and uses deterministic rounding rules.
- Recorded audio is absolute-time by default: source frames and start sample remain fixed when tempo changes. A future musical-time audio mode is deferred with time stretching.
- MIDI clips remain musical-time and follow tempo changes.
- All clip ranges are half-open `[start, end)` to remove boundary ambiguity.

### Thread Ownership And Data Flow

```text
JUCE message thread (sole ProjectModel writer)
  gesture -> DomainCommand -> validate -> apply -> Undo record -> revision++
                                      |
                                      +-> SnapshotBuilder(worker)
                                      +-> AutosaveCoordinator(worker)

SnapshotBuilder(revision N) -> immutable RuntimeSnapshot
  -> if N is current: SPSC publish -> audio callback swaps at buffer boundary
  -> if stale: discard

Audio callback -> bounded SPSC audio FIFO -> RecordingWriter thread -> .partial WAV
Audio callback -> lock-free health events -> Diagnostics drain on message thread
```

`ProjectModel`, `UndoManager` and UI selection stay on the message thread. File decode, waveform peaks, graph compilation, render and autosave use cancellable workers with revision tokens. Shutdown joins workers in a fixed order after transport and recording enter safe states.

### Project Persistence Contract

```text
project.composer/
  project.json               versioned manifest + integrity metadata
  Audio/<asset-id>.wav       immutable recorded/imported source
  Peaks/<asset-id>.peaks     disposable cache
  Autosave/<revision>.json   rotating recovery snapshots
  Recovery/<take-id>.journal recording finalization metadata
```

The save pipeline writes `project.json.tmp`, flushes it, parses and validates the temporary file, rotates the previous manifest, then atomically replaces the active manifest. Migrations are pure `vN -> vN+1` transforms tested from every supported version. Unknown newer fields are retained where possible; newer major schemas open read-only rather than being overwritten.

### Code Quality Review

**[P1] (confidence 8/10) Plan gap:** the original document used “service” names without dependency rules. The architecture above makes domain logic pure and turns device/file code into adapters. This prevents `ChordInsertionService`, `RhythmGenerationService` and UI handlers from duplicating validation or mutating notes differently.

**[P1] (confidence 8/10) Plan gap:** asynchronous workers lacked stale-result semantics. Every worker result now carries `projectId`, `entityId` and `revision`; mismatches discard the result. Cancellation is explicit and no worker owns UI objects.

**[P2] (confidence 7/10) Complexity risk:** generic `Manager` classes would collect branches for devices, projects and assets. Naming is role-based (`AudioDeviceController`, `ProjectRepository`, `AssetResolver`, `RecordingWriter`) and each public method returns a typed variant result. ### Command And Error Contracts

`scripts/dev.ps1` is the only convenience front door and supports `doctor`, `bootstrap`, `configure`, `build`, `test`, `run` and `package`. It accepts `-Verbose`, never installs system packages silently and exits nonzero with a stable error ID. `CMakePresets.json` remains the source of build/test configuration so IDE users and CI do not depend on opaque script state.

Every developer-facing failure uses:

```text
DX-CMAKE-002: CMake 3.x.y or newer is required; found 3.a.b.
Cause: the selected JUCE build preset requires the pinned minimum.
Fix: winget upgrade Kitware.CMake, reopen the terminal, then run .\scripts\dev.ps1 doctor.
Details: docs/TROUBLESHOOTING.md#dx-cmake-002
```

No message says only "build failed," "unknown audio error" or "see logs." Commands preserve a readable console summary and write detailed local logs beneath `build/diagnostics/`; they never upload source, project content, audio or paths automatically.

### Timeline Viewport Policy

- The arrangement timeline and contextual piano roll own independent horizontal scroll and zoom
  viewports. They share the transport playhead and absolute musical ticks, but navigating one view
  never moves or rescales the other unexpectedly.
- Selecting another MIDI clip resets the piano-roll scroll position to that clip's start while
  preserving the user's zoom level.
- The timeline is elastic and effectively unbounded to the right. Navigation grows in virtual
  chunks; adding, moving, resizing, pasting, or inserting notes beyond the stored clip end extends
  that clip in the same undoable command. Playback still ends at the last note rather than rendering
  an infinite silent tail. The implementation uses signed 64-bit ticks, so its technical ceiling is
  intentionally far beyond any practical song length rather than mathematical infinity.
- Trackpad `deltaX` pans horizontally. `Shift+wheel` is the mouse-wheel fallback.
- `Ctrl+wheel` and the native pinch gesture zoom the piano roll around the pointer. The musical
  tick under the pointer remains fixed except when the clip boundary must clamp the viewport.
- Note insertion floors to the start of the clicked grid cell. Note move and resize operations use
  nearest-grid snapping; holding `Alt` temporarily bypasses snapping.
